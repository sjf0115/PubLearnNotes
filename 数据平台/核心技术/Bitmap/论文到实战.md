## 1. 引言

Bit-Sliced Index（BSI，位切片索引）最早由 Patrick O'Neil 和 Dallan Quass 在 1997 年 ACM SIGMOD 会议上发表的经典论文《Improved Query Performance with Variant Indexes》[1] 中首次提出。该论文指出，在数据仓库的只读（read-mostly）环境中，可以利用更复杂的索引结构来加速 OLAP 查询。BSI 正是在这种背景下诞生的一种面向数值列的高效索引结构。

随后，Denis Rinfret、Patrick O'Neil 和 Elizabeth O'Neil 在 2001 年 SIGMOD 论文《Bit-Sliced Index Arithmetic》[2] 中进一步扩展了 BSI 的能力，引入了 BSI 算术运算（加法、减法等），使得 BSI 能够直接在索引层面完成多列间的数值计算。

本文将深入剖析 BSI 的核心算法，详细参考上述论文的理论基础，从数据结构定义、核心操作算法到复杂度分析，全面展示 BSI 的设计精髓。

## 2. 理论背景

### 2.1 数据仓库查询挑战

在数据仓库的星型模式（Star Schema）中，事实表通常包含大量的数值度量列（如销售额、数量等），分析查询往往需要对这些列执行如下操作：
- **范围过滤**：`WHERE dollar_sales > 100`
- **聚合计算**：`SUM(dollar_sales)`, `COUNT(*)`, `AVG(dollar_sales)`
- **排序取 TOP-K**：`ORDER BY dollar_sales DESC LIMIT K`

传统的 B-Tree 索引在这些场景下效率不高。O'Neil 在论文 [1] 中系统性地比较了多种变体索引（Variant Indexes），包括 Projection Index、Value-List Index 和 Bit-Sliced Index，证明了 BSI 在聚合查询和范围查询上的显著优势。

### 2.2 从 Bitmap Index 到 Bit-Sliced Index

传统的 Bitmap Index（位图索引）对列的每个不同取值都维护一个独立的 Bitmap。当列的基数（cardinality）很高时，需要维护大量的 Bitmap，存储开销巨大。

**Equality-Encoded Bitmap** 的局限性：
- 高基数列需要维护过多 Bitmap（基数为 N 则需要 N 个 Bitmap）
- 范围查询需要对多个 Bitmap 执行 OR 操作，代价高昂

**BSI 的核心思想**：将数值按位（bit）分解，每个二进制位对应一个 Bitmap（称为"切片"，Slice）。对于 N 位整数，只需要 N 个 Bitmap 即可表示任意范围的值。这将存储空间从 O(V) 降低到 O(log V)，其中 V 是值域大小。

### 2.3 论文中的形式化定义

根据 O'Neil 论文 [1] 中的定义：

> **Definition (Bit-Sliced Index)**: 给定一个具有 N 行的事实表的某个数值列 C，设 C 中的值最多需要 M 位二进制表示。BSI 由 M 个 Bitmap B[0], B[1], ..., B[M-1] 组成，其中 B[i] 的第 j 位为 1 当且仅当第 j 行的列值 C[j] 的第 i 个二进制位为 1。

形式化表示为：

\[B[i][j] = \lfloor C[j] / 2^i \rfloor \mod 2\]

此外，还需要一个 **Existence Bitmap（EBM）** 来记录哪些行在该列上有非空值。

## 3. BSI 数据结构

### 3.1 核心组成

BSI 由以下组件构成：

| 组件 | 描述 |
| :--- | :--- |
| slices[0..M-1] | M 个位切片 Bitmap，每个切片对应一个二进制位 |
| EBM | 存在位图，标记所有有效的行/Key |
| maxValue | 当前存储的最大值 |
| minValue | 当前存储的最小值 |
| M (sliceSize) | 切片数量，等于最大值的二进制位数 |

### 3.2 构造示例

假设有一组 Key-Value 数据（用户ID → 积分）：

| Key (user_id) | Value (score) | 二进制表示 |
| :--- | :--- | :--- |
| 1 | 13 | 01101 |
| 2 | 25 | 11001 |
| 3 | 7 | 00111 |
| 4 | 30 | 11110 |
| 5 | 18 | 10010 |

最大值为 30，需要 5 位二进制表示，因此 M = 5。构建过程为：对每个 Key-Value 对，将 Value 的第 i 位为 1 的 Key 添加到 slices[i] 中。

构造结果：

| 切片 | 包含的 Key 集合 | 含义 |
| :--- | :--- | :--- |
| slices[0] | {1, 2, 3} | Value 的第 0 位为 1 的 Key |
| slices[1] | {3, 4, 5} | Value 的第 1 位为 1 的 Key |
| slices[2] | {1, 3} | Value 的第 2 位为 1 的 Key |
| slices[3] | {1, 2, 4} | Value 的第 3 位为 1 的 Key |
| slices[4] | {2, 4, 5} | Value 的第 4 位为 1 的 Key |
| EBM | {1, 2, 3, 4, 5} | 所有存在的 Key |

**验证**：Key=1 的值为 13（二进制 01101），第 0、2、3 位为 1，因此 Key=1 出现在 slices[0]、slices[2]、slices[3] 中，正确。

### 3.3 空间复杂度分析

设事实表有 N 行，数值列最大值为 V：
- 传统 Equality-Encoded Bitmap Index：需要 V 个 Bitmap，每个 N 位，总空间 O(N·V)
- BSI：需要 log₂(V) + 1 个 Bitmap，每个 N 位，总空间 O(N·log V)

当 V 很大时（如百万级销售额），BSI 的空间优势极为明显。

## 4. 核心算法

### 4.1 聚合查询算法（SUM）

O'Neil 论文 [1] 中的 Algorithm 3.2 给出了使用 BSI 计算聚合 SUM 的方法。这是 BSI 最具代表性的优势之一。

**问题定义**：给定一个候选行集 F（通常由 WHERE 子句筛选得到的 Bitmap），计算这些行上数值列 C 的总和。

**算法思想**：利用二进制位值的加权求和性质。对于 M 位的 BSI：

\[SUM = \sum_{i=0}^{M-1} 2^i \times |B[i] \cap F|\]

其中 \(|B[i] \cap F|\) 表示切片 B[i] 与候选集 F 的交集基数（即同时满足条件且第 i 位为 1 的行数）。

**伪代码**：
```
Algorithm SUM_BSI(F, B[0..M-1]):
    Input: F - 候选行集 Bitmap
           B[0..M-1] - BSI 切片数组
    Output: SUM 值

    sum ← 0
    for i ← 0 to M-1 do
        count ← AND_COUNT(B[i], F)   // B[i] ∩ F 的基数
        sum ← sum + 2^i × count
    end for
    return sum
```

**Java 实现**：
```java
public long sum(RoaringBitmap foundSet) {
    if (foundSet == null || foundSet.isEmpty()) {
        return 0L;
    }
    long sum = 0;
    for (int i = 0; i < sliceSize; i++) {
        long sliceWeight = 1L << i;  // 2^i
        sum += sliceWeight * RoaringBitmap.andCardinality(slices[i], foundSet);
    }
    return sum;
}
```

**复杂度分析**：
- 时间复杂度：O(M × T_AND)，其中 T_AND 是两个 Bitmap 执行 AND-COUNT 的时间
- 相比传统方法逐行累加的 O(|F|)，当使用压缩 Bitmap（如 RoaringBitmap）时，BSI 方法通常快得多
- 论文 [1] 中的性能对比表明：BSI SUM 算法在大数据集上比基于 B-Tree 或 Projection Index 的方案有 2-5 倍的性能提升

### 4.2 范围查询算法（Range Query）

范围查询是 BSI 的另一核心能力，对应 O'Neil 论文 [1] 中的 Algorithm 3.3 和 Algorithm 3.4。

**问题定义**：给定一个目标值 V，找出所有满足 C > V、C < V 或 C = V 的行集合。

**核心思想**：从最高位到最低位逐位比较，逐步将候选集分流到 GT（大于）、LT（小于）和 EQ（等于）三个集合中。这个过程类似于二进制数的逐位比较。

**算法推导**：

设目标值 V 的二进制表示为 \(v_{M-1} v_{M-2} \ldots v_1 v_0\)。初始时，所有行都是"可能相等"的候选者（EQ = EBM）。从最高位 M-1 开始逐位扫描：

- **当 \(v_i = 1\) 时**：
  - EQ 中第 i 位为 0 的行：其值在前缀相同的情况下，第 i 位比 V 小，必然 < V → 转入 LT
  - EQ 中第 i 位为 1 的行：到目前为止与 V 完全一致 → 留在 EQ

- **当 \(v_i = 0\) 时**：
  - EQ 中第 i 位为 1 的行：其值在前缀相同的情况下，第 i 位比 V 大，必然 > V → 转入 GT
  - EQ 中第 i 位为 0 的行：到目前为止与 V 完全一致 → 留在 EQ

**伪代码**：
```
Algorithm RANGE_BSI(V, B[0..M-1], EBM):
    Input: V - 目标比较值
           B[0..M-1] - BSI 切片数组
           EBM - 存在位图
    Output: GT, EQ, LT 三个行集 Bitmap

    GT ← ∅
    LT ← ∅
    EQ ← EBM

    for i ← M-1 downto 0 do
        v_i ← (V >> i) AND 1    // V 的第 i 位
        if v_i = 1 then
            LT ← LT OR (EQ AND_NOT B[i])  // EQ中第i位为0的 → LT
            EQ ← EQ AND B[i]              // EQ中第i位为1的留下
        else
            GT ← GT OR (EQ AND B[i])      // EQ中第i位为1的 → GT
            EQ ← EQ AND_NOT B[i]          // EQ中第i位为0的留下
        end if
    end for

    return (GT, EQ, LT)
```

**Java 实现**：
```java
public RoaringBitmap[] rangeQuery(int value) {
    RoaringBitmap GT = new RoaringBitmap();
    RoaringBitmap LT = new RoaringBitmap();
    RoaringBitmap EQ = this.ebm;

    for (int i = sliceSize - 1; i >= 0; i--) {
        int bit = (value >> i) & 1;
        if (bit == 1) {
            // V的第i位为1: EQ中第i位为0的行 < V
            LT = RoaringBitmap.or(LT, RoaringBitmap.andNot(EQ, slices[i]));
            EQ = RoaringBitmap.and(EQ, slices[i]);
        } else {
            // V的第i位为0: EQ中第i位为1的行 > V
            GT = RoaringBitmap.or(GT, RoaringBitmap.and(EQ, slices[i]));
            EQ = RoaringBitmap.andNot(EQ, slices[i]);
        }
    }
    return new RoaringBitmap[]{GT, EQ, LT};
}
```

**正确性证明**：

对于任意行 j，设 C[j] 的二进制表示为 \(c_{M-1} c_{M-2} \ldots c_0\)。考虑从高位到低位的比较过程：
- 在第 k 位首次出现 \(c_k \neq v_k\) 时（之前所有高位都相等）：
  - 若 \(c_k = 1, v_k = 0\)，则 C[j] > V（因为 \(2^k > \sum_{i=0}^{k-1} 2^i\)），行 j 被转入 GT
  - 若 \(c_k = 0, v_k = 1\)，则 C[j] < V，行 j 被转入 LT
- 若所有位都相等，则行 j 留在 EQ

这与二进制数大小比较的定义完全一致，因此算法正确。

**复杂度分析**：
- 时间复杂度：O(M × T_BITMAP_OP)，其中 M 为切片数（通常 20-30 位足以覆盖大多数数值列）
- 每次迭代执行常数次 Bitmap 操作（AND、OR、ANDNOT）
- 总计 3M 次 Bitmap 操作

### 4.3 查询组合

基于 Range Query 算法得到的三个集合，可以组合出所有比较操作：

| 查询条件 | 返回结果 |
| :--- | :--- |
| C = V | EQ |
| C ≠ V | EBM - EQ (即 ANDNOT) |
| C > V | GT |
| C < V | LT |
| C ≥ V | GT ∪ EQ |
| C ≤ V | LT ∪ EQ |
| L ≤ C ≤ U | GE(L) ∩ LE(U) |

### 4.4 COUNT 算法

COUNT 操作本质上是 SUM 的简化形式。论文 [1] 中指出，计算满足条件的行数只需返回候选集的基数：

```
Algorithm COUNT_BSI(F):
    return |F|    // 即 Bitmap F 的 popcount
```

这在 RoaringBitmap 中可以通过 `getLongCardinality()` 在 O(1) 时间内完成（因为 RoaringBitmap 维护了基数缓存）。

### 4.5 MAX/MIN 算法

O'Neil 论文 [1] 的 Algorithm 3.5 给出了使用 BSI 计算 MAX 的方法。

**MAX 算法思想**：从最高位到最低位贪心选择。对于每个位 i，尝试将候选集限定为该位为 1 的行。如果限定后候选集非空，则保留；否则放弃该位。

**伪代码**：
```
Algorithm MAX_BSI(F, B[0..M-1]):
    Input: F - 候选行集 Bitmap
           B[0..M-1] - BSI 切片数组
    Output: 候选集中的最大值

    candidates ← F
    for i ← M-1 downto 0 do
        temp ← candidates AND B[i]
        if temp ≠ ∅ then
            candidates ← temp
        end if
    end for
    // candidates 中任取一行，读取其值即为最大值
    return getValue(first(candidates))
```

**Java 实现**：
```java
public int maxValue(RoaringBitmap foundSet) {
    RoaringBitmap candidates = RoaringBitmap.and(foundSet, ebm);
    if (candidates.isEmpty()) {
        return -1;
    }
    for (int i = sliceSize - 1; i >= 0; i--) {
        RoaringBitmap temp = RoaringBitmap.and(candidates, slices[i]);
        if (!temp.isEmpty()) {
            candidates = temp;
        }
    }
    return getValue(candidates.first());
}
```

**MIN 算法**与 MAX 对称，从最高位开始，尝试将候选集限定为该位为 0 的行：

```java
public int minValue(RoaringBitmap foundSet) {
    RoaringBitmap candidates = RoaringBitmap.and(foundSet, ebm);
    if (candidates.isEmpty()) {
        return -1;
    }
    for (int i = sliceSize - 1; i >= 0; i--) {
        RoaringBitmap temp = RoaringBitmap.andNot(candidates, slices[i]);
        if (!temp.isEmpty()) {
            candidates = temp;
        }
    }
    return getValue(candidates.first());
}
```

**复杂度**：O(M × T_BITMAP_OP)，只需遍历一次所有切片。

### 4.6 TOP-K 算法

论文 [2] 进一步提出了基于 BSI 的 TOP-K 查询算法。其核心思想是对 MAX 算法的推广：不再贪心地仅保留一个分支，而是维护一个"至少 K 行"的候选集。

**伪代码**：
```
Algorithm TOPK_BSI(F, B[0..M-1], K):
    Input: F - 候选行集, K - 取前K个
    Output: 值最大的K行

    candidates ← F
    for i ← M-1 downto 0 do
        temp ← candidates AND B[i]
        if |temp| >= K then
            candidates ← temp
        end if
    end for
    return candidates
```

当 |temp| ≥ K 时，说明仅保留第 i 位为 1 的行也足够取出 K 个最大值；否则需要保留全部候选，等待低位进一步区分。

## 5. BSI 算术运算

### 5.1 论文背景

Rinfret 等人在论文 [2]《Bit-Sliced Index Arithmetic》中指出，数据仓库查询常常需要对多列进行算术组合，例如：
```sql
SELECT SUM(price * quantity) FROM sales WHERE ...
```

传统方法需要先逐行计算 `price * quantity`，再聚合。BSI 算术使得这些运算可以直接在索引层面高效完成。

### 5.2 BSI 加法

给定两个 BSI：A[0..P-1] 和 B[0..Q-1]，计算逐行加和 C = A + B 的 BSI。

**算法核心**：模拟二进制全加器（Full Adder），逐位相加并处理进位。

```
Algorithm BSI_ADD(A[0..P-1], B[0..Q-1]):
    Input: 两个 BSI
    Output: 和 BSI C[0..max(P,Q)]

    carry ← ∅    // 进位 Bitmap
    M ← max(P, Q)

    for i ← 0 to M-1 do
        a_i ← A[i] if i < P else ∅
        b_i ← B[i] if i < Q else ∅

        // 全加器: sum = a XOR b XOR carry
        //         new_carry = (a AND b) OR (a AND carry) OR (b AND carry)
        C[i] ← a_i XOR b_i XOR carry
        carry ← (a_i AND b_i) OR (a_i AND carry) OR (b_i AND carry)
    end for

    if carry ≠ ∅ then
        C[M] ← carry
    end if
    return C
```

**关键洞察**（论文 [2] 中指出）：BSI 加法的复杂度为 O(M × T_BITMAP_OP)，与行数 N 无关（Bitmap 操作本身与 N 相关，但利用压缩 Bitmap 可以非常高效）。这比逐行相加 O(N) 的方法在合适条件下更有优势。

### 5.3 BSI 减法

类似于加法，BSI 减法模拟二进制减法器（借位逻辑）：

```
Algorithm BSI_SUB(A[0..P-1], B[0..Q-1]):
    // 计算 C = A - B (假设 A >= B 对所有行成立)
    borrow ← ∅

    for i ← 0 to M-1 do
        a_i ← A[i] if i < P else ∅
        b_i ← B[i] if i < Q else ∅

        C[i] ← a_i XOR b_i XOR borrow
        borrow ← (NOT a_i AND b_i) OR (NOT a_i AND borrow) OR (b_i AND borrow)
    end for
    return C
```

### 5.4 应用场景

BSI 算术运算使得以下查询可以高效执行：
- `SUM(A + B)`：先执行 BSI_ADD 得到 C，再对 C 执行 SUM 算法
- `SUM(A * constant)`：通过移位和加法实现
- 多属性评分排序：如 `score = 0.3*A + 0.7*B`

## 6. 性能分析与对比

### 6.1 O'Neil 论文中的性能对比

论文 [1] 在 Section 4 中通过代价模型比较了四种方案处理聚合查询 `SELECT AGG(C) FROM T WHERE condition` 的 I/O 成本：

| 方案 | SUM 查询代价 | 范围查询代价 |
| :--- | :--- | :--- |
| Sequential Scan | O(N/blocking_factor) | O(N/blocking_factor) |
| Projection Index | O(N/blocking_factor) | O(N/blocking_factor) |
| Value-List Index | O(V × bitmap_size) | O(range × bitmap_size) |
| **Bit-Sliced Index** | **O(M × bitmap_size)** | **O(M × bitmap_size)** |

其中 M = ⌈log₂V⌉，通常远小于 V 和 N。

### 6.2 各操作复杂度总结

| 操作 | 时间复杂度 | Bitmap 操作次数 |
| :--- | :--- | :--- |
| SUM | O(M) | M 次 AND-COUNT |
| Range (GT/LT/EQ) | O(M) | 3M 次 AND/OR/ANDNOT |
| MAX / MIN | O(M) | M 次 AND/ANDNOT |
| TOP-K | O(M) | M 次 AND + COUNT |
| BSI ADD | O(M) | ~5M 次 XOR/AND/OR |
| 单行 GET | O(M) | M 次 CONTAINS |
| 单行 PUT | O(M) | M 次 ADD/REMOVE |

> 注：这里的 O(M) 是指 Bitmap 操作的次数，每次 Bitmap 操作本身的复杂度取决于具体实现（如 RoaringBitmap 通常为 O(N/容器大小) 级别）。

### 6.3 BSI vs 其他索引方案

| 维度 | B-Tree | Equality Bitmap | BSI |
| :--- | :--- | :--- | :--- |
| 存储空间 | O(N) | O(N × V) | O(N × log V) |
| 精确查找 | O(log N) | O(1) bitmap | O(M) bitmap ops |
| 范围查询 | O(log N + R) | O(V) OR ops | O(M) bitmap ops |
| SUM 聚合 | O(R) 逐行累加 | 不支持 | O(M) bitmap ops |
| 高基数支持 | 好 | 差 | 好 |
| 适用场景 | OLTP | 低基数过滤 | OLAP 聚合与范围 |

## 7. 实现优化策略

### 7.1 压缩 Bitmap 的选择

论文 [1] 发表时主要使用朴素 Bitmap，现代实现通常采用压缩 Bitmap：
- **RoaringBitmap**：自适应容器策略（Array/Bitmap/Run），在稀疏和稠密数据上都有优秀表现
- **WAH (Word-Aligned Hybrid)**：早期压缩方案，对连续相同位模式压缩效果好
- **EWAH**：WAH 的改进版本

RoaringBitmap 是当前 BSI 实现的最佳选择，其 AND/OR/XOR 操作在压缩状态下即可完成，无需解压。

### 7.2 负数处理

原始 BSI 设计仅支持非负整数。处理负数的常见方案：
- **偏移编码**：将所有值加上一个偏移量，使最小值变为 0
- **符号位分离**：用额外的一个 Bitmap 存储符号位，正负数分别处理后合并

### 7.3 浮点数处理

对于浮点数值列，通常的策略是：
- **定点化**：乘以 10^k 转为整数（如保留两位小数则乘以 100）
- **适用于精度可控的金融/度量数据**

### 7.4 并行化

BSI 的各切片之间天然独立，可以并行处理：
- SUM 操作中，各切片的 AND-COUNT 可以并行执行
- Range Query 中的逐位处理有数据依赖（EQ 集合逐步缩小），较难并行化
- BSI 算术运算中的进位链也限制了并行度

## 8. BSI 在现代系统中的应用

### 8.1 用户画像系统

BSI 在用户画像平台中有广泛应用：
- **标签存储**：用户年龄、消费金额等数值标签用 BSI 存储
- **圈人查询**：`年龄 >= 18 AND 年龄 <= 35` 直接用 Range Query 实现
- **聚合分析**：`SUM(消费金额) WHERE 城市 = '北京'`

### 8.2 时序数据库

### 9.2 Range Query 验证

查询 Value > 18 的用户，即 V = 18（二进制 10010）：

初始：EQ = {1,2,3,4,5}, GT = ∅, LT = ∅

**第 4 位**（v₄ = 1）：
- LT = LT ∪ (EQ \ slices[4]) = ∅ ∪ ({1,2,3,4,5} \ {2,4,5}) = {1,3}
- EQ = EQ ∩ slices[4] = {1,2,3,4,5} ∩ {2,4,5} = {2,4,5}

**第 3 位**（v₃ = 0）：
- GT = GT ∪ (EQ ∩ slices[3]) = ∅ ∪ ({2,4,5} ∩ {1,2,4}) = {2,4}
- EQ = EQ \ slices[3] = {2,4,5} \ {1,2,4} = {5}

**第 2 位**（v₂ = 0）：
- GT = GT ∪ (EQ ∩ slices[2]) = {2,4} ∪ ({5} ∩ {1,3,4}) = {2,4} ∪ ∅ = {2,4}
- EQ = EQ \ slices[2] = {5} \ {1,3,4} = {5}

**第 1 位**（v₁ = 1）：
- LT = LT ∪ (EQ \ slices[1]) = {1,3} ∪ ({5} \ {3,4,5}) = {1,3} ∪ ∅ = {1,3}
- EQ = EQ ∩ slices[1] = {5} ∩ {3,4,5} = {5}

**第 0 位**（v₀ = 0）：
- GT = GT ∪ (EQ ∩ slices[0]) = {2,4} ∪ ({5} ∩ {1,2,3}) = {2,4} ∪ ∅ = {2,4}
- EQ = EQ \ slices[0] = {5} \ {1,2,3} = {5}

- GT = {2, 4}，即 user 2（score=25）和 user 4（score=30）> 18
- EQ = {5}，即 user 5（score=18）= 18
- LT = {1, 3}，即 user 1（score=13）和 user 3（score=7）< 18

验证正确。

## 10. 总结

Bit-Sliced Index（BSI）是一种面向数值列的高效索引结构，其核心优势在于：

1. **空间高效**：仅需 O(log V) 个 Bitmap 即可表示值域为 V 的数值列，相比 Equality-Encoded Bitmap 的 O(V) 有指数级的空间缩减。

2. **查询高效**：所有核心操作（SUM、Range、MAX/MIN、TOP-K）的复杂度均为 O(M)，其中 M 通常为 20-30，与数据行数 N 无关。

3. **可组合性强**：论文 [2] 提出的 BSI 算术运算使得多列间的加减运算可以在索引层面直接完成，避免了物化中间结果。

4. **与压缩 Bitmap 天然契合**：现代 RoaringBitmap 等压缩位图的高效位运算使得 BSI 的实际性能远超理论预期。

BSI 从提出至今已有近 30 年的历史，但其设计思想依然深刻地影响着现代 OLAP 系统和用户画像平台的核心架构。
